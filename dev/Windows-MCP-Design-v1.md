# Windows 兼容 Claude / Cursor / Codex 的 MCP 设计文档 v1

## 1. 项目目标

本项目面向 Windows 平台，构建一个通用桌面智能体执行平台。系统通过 MCP（Model Context Protocol）向 Claude、Codex、Cursor 等客户端暴露标准工具，提供桌面感知、应用操控、跨应用协同、长任务编排和安全控制能力。

核心目标：

1. 接受自然语言命令，并能自主规划、决策和执行。
2. 在 Windows 上稳定控制已安装应用、系统窗口和文件资源。
3. 支持长任务、断点恢复和后台持续运行。
4. 保证高交互精度、强稳定性与可审计安全。
5. 对 Claude / Cursor / Codex 保持统一接入方式。

---

## 2. 全局开发规则

本项目采用模块化开发，先明确具体架构、技术实现和每个模块在架构中的位置和作用，再进行单个模块开发。模块之间尽量解耦。注意每个小模块开发完成后，都要进行实际测试，确保正常运行，防止项目后期崩溃。
dev文件夹才是本次开发的主文件夹，Windows-MCP-main源码文件夹只是部分后端功能的实现，dev里面会实现Windows-MCP-main里面没有或效果不佳的更强大，更健壮，更高效，更创新的功能。
补充约定：

- 后续开发统一在独立 dev 文件夹中进行。
- 如需参考 `Windows-MCP-main` 的代码，只能复制到新文件夹后再开发，不直接修改原仓库。
- 每次新增或修改模块，都必须同步补充测试计划。
- 每个模块必须先定义接口，再实现内部逻辑。
- 优先保证稳定性与可恢复性，其次才追求功能扩展。

## 3. 设计原则

### 3.1 分层解耦

系统分为以下层级：

- 客户端接入层：Claude / Cursor / Codex 等。
- MCP 协议层：统一工具与资源定义。
- 任务编排层：计划、拆解、调度、恢复。
- 感知层：截图、UIA 树、OCR、窗口状态。
- 执行层：鼠标、键盘、应用启动、进程控制、文件操作。
- 安全层：权限、审计、确认、白名单、沙箱。

### 3.2 工具优先，模型独立

平台不依赖某一个模型或某一个客户端。模型只负责推理与规划，真正执行交给平台工具层。

### 3.3 闭环控制

每个动作必须满足：

1. 观察当前状态。
2. 决策下一步动作。
3. 执行动作。
4. 验证执行结果。
5. 必要时重试、回滚或请求确认。

### 3.4 安全默认开启

默认启用安全策略，敏感操作必须经过确认或策略授权。

## 4. 系统总体架构

### 4.1 架构图

```text
Claude / Cursor / Codex
            |
            v
      MCP Client Adapter
            |
            v
        MCP Server
            |
   -----------------------
   |         |           |
   v         v           v
Planner   Perception   Safety Gate
   |         |           |
   v         v           v
        Task Orchestrator
                |
                v
         Windows Executor
                |
   -----------------------------
   |       |        |          |
   v       v        v          v
 UIA     Input    App/Proc   FS/Clipboard
   |
   v
 OCR / Screenshot / Window State
```

### 4.2 架构解释

- **MCP Server**：对外提供统一工具。
- **Planner**：把自然语言转成可执行步骤。
- **Perception**：识别当前桌面与应用状态。
- **Task Orchestrator**：管理长任务、状态机和恢复。
- **Windows Executor**：调用 Windows 原生能力执行动作。
- **Safety Gate**：对高风险操作进行限制和审计。

## 5. 模块设计

### 5.1 MCP 接入层

职责：

- 暴露标准工具。
- 兼容 stdio / SSE / streamable HTTP。
- 适配 Claude、Cursor、Codex 等客户端。
- 负责参数校验和会话管理。

建议工具类型：

- `desktop.snapshot`
- `desktop.screenshot`
- `desktop.windows`
- `input.click`
- `input.drag`
- `input.type`
- `input.shortcut`
- `app.launch`
- `app.focus`
- `app.close`
- `fs.read`
- `fs.write`
- `clipboard.get`
- `clipboard.set`
- `task.plan`
- `task.run`
- `task.status`
- `security.request_confirmation`

### 5.2 任务编排层

职责：

- 将复杂目标拆成步骤。
- 维护步骤依赖关系。
- 处理重试、超时与恢复。
- 支持后台执行与任务挂起。

建议设计：

- `TaskPlan`
- `TaskStep`
- `TaskCheckpoint`
- `TaskResult`
- `TaskState`

关键能力：

- step-by-step 执行。
- 支持人工介入后继续。
- 支持 job id 查询。
- 支持中断恢复。

### 5.3 感知层

职责：

- 截图采集。
- UI Automation 树读取。
- OCR 文本识别。
- 活动窗口和进程识别。
- 多显示器和 DPI 处理。

推荐定位优先级：

1. UIA 控件定位。
2. 文本和语义匹配。
3. OCR 定位。
4. 像素坐标兜底。

### 5.4 执行层

职责：

- 鼠标点击、拖拽、移动。
- 键盘输入、快捷键。
- 应用启动、切换、关闭。
- 进程列表与终止。
- 剪贴板读写。
- 文件和注册表操作（按策略控制）。

执行策略：

- 优先调用原生 API。
- 失败时切换到模拟输入。
- 每步都要验证结果。

### 5.5 安全层

职责：

- 敏感操作二次确认。
- 文件与应用白名单。
- 高风险命令阻断。
- 审计日志。
- 只读模式与自动化模式切换。

建议安全级别：

- `observe-only`
- `controlled`
- `full-automation`

### 5.6 状态管理层

职责：

- 保存任务上下文。
- 保存最近截图摘要。
- 保存操作历史。
- 保存恢复点。
- 保存失败原因。

建议持久化字段：

- `task_id`
- `step_index`
- `window_focus`
- `last_observation`
- `retry_count`
- `safety_decision`

## 6. Windows 底层技术路线

### 6.1 推荐主路径

- UI Automation（UIA）
- Win32 / User32
- SendInput
- Window handle 操作
- 进程管理（psutil / 原生接口）
- 截图引擎（dxcam / mss / pillow）
- OCR 模块

### 6.2 辅助路径

- PowerShell
- AutoHotkey
- COM / COMtypes
- Registry 访问
- 文件系统工具

### 6.3 视觉增强

当 UIA 读取不到或控件不标准时，引入：

- OCR + 坐标定位。
- 视觉模型识别控件位置。
- 组合定位策略。

## 7. 兼容 Claude / Cursor / Codex 的方式

### 7.1 统一接入策略

只做一个标准 MCP Server，尽量不做客户端特化逻辑。

### 7.2 客户端接入方式

- Claude Desktop：stdio 或 streamable HTTP。
- Cursor：MCP server 配置。
- Codex：MCP server 配置。

### 7.3 统一能力边界

所有客户端都看到同一套工具与资源，避免碎片化实现。

---

## 8. 长任务运行设计

### 8.1 任务生命周期

1. 创建任务。
2. 规划任务。
3. 执行步骤。
4. 存储 checkpoint。
5. 失败重试或等待用户确认。
6. 完成任务。
7. 记录审计结果。

### 8.2 恢复能力

- 断电后可恢复。
- 客户端断连后可继续。
- 任务失败可从最近 checkpoint 恢复。

---

## 9. 高精度交互设计

### 9.1 元素定位策略

每个元素应包含：

- 文本。
- role。
- bounds。
- screen id。
- confidence。
- source。

### 9.2 定位优先级

- 先控件语义。
- 再视觉识别。
- 最后坐标点击。

### 9.3 交互纠错

如果点击后状态不变：

- 自动重试。
- 改用备用定位方式。
- 必要时截图并请求人工确认。

---

## 10. 稳定性设计

### 10.1 失败处理

- 超时。
- 重试。
- 降级。
- 回滚。
- 中止任务。

### 10.2 健康检查

- 当前前台窗口是否存在。
- UIA 树是否可读。
- 截图是否正常。
- 工具服务是否存活。

## 11. 跨应用协同设计

### 11.1 协同方式

- 应用 A 输出数据。
- 中间层标准化数据。
- 应用 B 读取数据。
- 通过剪贴板、文件、窗口、进程联动。

### 11.2 典型场景

- 邮件 -> 表格 -> 报告 -> 再发送。
- 浏览器 -> 文档 -> IDE -> 测试工具。
- 下载文件 -> 解压 -> 编辑 -> 上传。

### 12.1 安全策略

## 15. 结论

该平台的本质是一个面向 Windows 的通用桌面智能体执行系统。其核心价值不在于单次点击，而在于“观察—规划—执行—验证”的闭环能力。通过 MCP 标准协议对外兼容 Claude、Cursor、Codex，通过 Windows 原生自动化能力对内实现高精度和高稳定性，再配合状态管理、安全控制和跨应用协同，即可形成一个可持续演进的桌面智能体平台。
