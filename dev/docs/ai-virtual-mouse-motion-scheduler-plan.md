## 1. 背景与目标

当前桌面自动化的体验是：

- 模型思考期间界面静默
- 执行动作时鼠标瞬间跳转、点击
- 用户看不到“AI 在怎么操作”
- 整体体验更像黑盒自动化，而不是可观察的智能助手

我们希望引入一套**可见的 AI 虚拟鼠标**，让用户能看到：

- 鼠标为什么去那里
- 鼠标是沿什么路径移动的
- 每一步动作什么时候开始、什么时候结束
- 哪一步失败、哪一步被取消、哪一步验证成功

### 目标体验

用户看到的不是“瞬移点击”，而是：

1. 模型决定下一步动作
2. 虚拟鼠标沿自然路径移动到目标
3. 到达目标前短暂停顿
4. 真实输入执行点击/拖拽
5. 系统验证结果
6. 进入下一步动作

---

## 2. 方案选择

采用 **方案 B：动作调度器 + 路径动画 + 真实输入同步**。

### 核心原则

- **真实输入仍然负责执行**：保证兼容性和稳定性
- **虚拟鼠标只负责可视化**：让用户看到 AI 的意图和过程
- **执行前先动画，执行时同步，执行后验证**：形成完整闭环

### 为什么这是最现实的方案

相比“纯动画层”或“纯视觉替身”，方案 B 的关键优势是：

- 不破坏现有 Windows 桌面输入可靠性
- 便于逐步演进，不需要一次性重写底层执行器
- 兼容真实应用、窗口、控件、拖拽、快捷键等场景
- 可扩展到调试回放、失败重试、任务中断、日志分析

## 3. 现有实现概览

当前实现中，`Executor` 直接调用 backend 执行输入动作：

- `move(x, y)` 直接注入鼠标移动
- `click(x, y)` 直接注入点击
- `drag(start, end)` 直接注入拖拽
- `type_text(...)` 直接发送键盘输入
- `shortcut(...)` 直接发送快捷键
- `focus_window(...)` / `switch_window(...)` 直接切换窗口

这意味着当前没有：

- 虚拟鼠标状态
- 动作状态机
- 路径动画层
- overlay 渲染层
- 动作时间线（timeline）
- 中断、取消、重试框架

要实现“可见 AI 虚拟鼠标”，必须新增一层动作编排系统，而不是仅在原有 `move()` 上加动画。

---

## 4. 总体架构

建议拆成五层：

1. **Planner / Model Layer**
2. **Action Scheduler Layer**
3. **Virtual Cursor Layer**
4. **Overlay Renderer Layer**
5. **Executor / Verification Layer**

### 4.1 Planner / Model Layer

模型只输出高层意图，不直接关心鼠标动画。

例如：

- `move_to(x, y)`
- `click(x, y)`
- `drag(start, end)`
- `scroll(direction)`
- `focus_window(name)`
- `type_text(text)`

职责：

- 决定下一步做什么
- 提供目标坐标、动作类型、语义理由
- 不参与路径生成与渲染

### 4.2 Action Scheduler Layer

负责把高层动作转换成可执行流程：

- `planning`
- `animating`
- `executing`
- `verifying`
- `verified`
- `failed`
- `cancelled`

职责：

- 生成路径
- 控制动画节奏
- 控制真实输入执行时机
- 接管中断、重试、取消
- 记录动作时间线

### 4.3 Virtual Cursor Layer

负责维护虚拟鼠标的视觉状态：

- 当前坐标
- 当前按下状态
- 当前目标点
- 当前路径点列表
- 当前动作 ID
- 当前速度参数

职责：

- 作为 overlay 的状态源
- 作为 scheduler 的推进状态
- 在 UI 上表现“AI 正在操作”

### 4.4 Overlay Renderer Layer

负责在桌面上绘制透明悬浮层，展示：

- 鼠标指针
- 轨迹尾巴
- 点击波纹
- 抓取/拖拽状态
- 当前目标高亮

职责：

- 接收虚拟鼠标状态并渲染
- 不参与真实输入
- 不阻塞系统交互

### 4.5 Executor / Verification Layer

继续沿用现有真实输入逻辑，但由调度器控制执行时机，并在执行后做状态验证。

职责：

- 执行真实鼠标和键盘输入
- 执行窗口操作
- 提供结果给验证层
- 保持底层稳定性

---

## 5. 推荐的执行流程

以一次点击为例，完整流程如下：

1. 模型决定点击目标 `(x, y)`
2. 调度器生成路径
3. overlay 显示虚拟鼠标移动
4. 到达目标前短暂停顿
5. 执行真实系统点击
6. 读取系统状态和结果
7. 验证点击是否命中目标
8. 进入下一步动作

---

## 6. 状态机设计

建议每个动作都具备明确状态：

- `planned`
- `animating`
- `executing`
- `verifying`
- `verified`
- `failed`
- `cancelled`

### 状态转移示例

```text
planned -> animating -> executing -> verifying -> verified
```

失败路径：

```text
planned -> animating -> executing -> failed
planned -> animating -> cancelled
planned -> animating -> executing -> verifying -> target_mismatch
```

## 7. Virtual Cursor State 设计

建议维护如下状态结构：

```ts
type VirtualCursorState = {
  visible: boolean;
  x: number;
  y: number;
  pressed: boolean;
  currentActionId?: string;
  targetX?: number;
  targetY?: number;
  path: Array<{ x: number; y: number; t: number }>;
  velocity: number;
  state: "idle" | "planning" | "animating" | "executing" | "verifying";
};
```

### 作用

- 维护当前虚拟鼠标位置
- 保存当前轨迹
- 记录动作执行状态
- 供 overlay 渲染使用

## 8. 路径生成策略

不要使用直线匀速移动，视觉上会很机械。

### 推荐路径算法

#### 8.1 贝塞尔曲线
适合普通移动，简单、自然、实现成本低。

#### 8.2 分段样条曲线
适合长距离移动，更像人手轨迹。

#### 8.3 带轻微扰动的平滑路径
适合增强“像真人”的感觉，但扰动必须克制。

### 路径生成原则

- 起步慢
- 中段快
- 接近目标时减速
- 不要直线匀速
- 拖拽轨迹比普通移动更稳定

---

## 9. 动作类型的执行策略

### 9.1 点击 click

#### 流程

1. 生成目标点路径
2. 虚拟鼠标移动过去
3. 到点停顿 60~120ms
4. 真实执行 click
5. 点击波纹反馈
6. 验证结果

#### 推荐时长

- 短距离：200~300ms
- 长距离：400~700ms

---

### 9.2 拖拽 drag

#### 流程

1. 先移动到起点
2. 按下
3. 沿轨迹拖动到终点
4. 松开
5. 短暂停顿
6. 验证拖拽结果

#### 推荐时长

- 总动画：600~1200ms
- 更长的拖拽：最多到 1500ms

---

### 9.3 移动 move

#### 用途

- 提示下一步意图
- Hover 预览
- 显示 AI 正在观察哪里

#### 推荐时长

- 150~500ms
- 远距离可稍长，但不要太慢

---

### 9.4 多步连续任务

例如：

- 打开设置
- 点击网络
- 打开 Wi‑Fi
- 切换开关

#### 推荐做法

- 步与步之间保留 150~300ms 的观察停顿
- 虚拟鼠标停在目标上，让用户感知 AI 的意图
- 保留短暂的“思考 + 观察 + 执行”节奏

---

## 10. Overlay 渲染层设计

### 推荐实现

在 Windows 上采用一个**透明、置顶、无边框、鼠标穿透**的悬浮窗口来绘制虚拟鼠标。

### 视觉元素

- 鼠标指针本体
- 最近轨迹尾巴
- 点击波纹
- 拖拽按下状态
- 目标点高亮

### 体验建议

- 轨迹尾巴只保留最近 300~800ms
- 轨迹逐渐淡出
- 点击时显示圆形波纹并快速消失
- 不要过度炫技，保持克制和清晰

---

## 11. 真实输入与虚拟输入同步

### 核心原则

- 虚拟鼠标负责“看见”
- 真实输入负责“执行”
- 两者要时间对齐，但不需要逐帧完全一致

### 推荐同步方式

#### 方式 1：先动画，后真实执行
最简单、最稳，推荐优先实现。

#### 方式 2：动画与真实输入几乎同步
更高级，但实现复杂，建议后期再做。

---

## 12. 中断、失败、重试、取消

方案 B 必须支持这些能力。

### 12.1 中断

当用户手动移动鼠标、点击别处或按 Esc 时：

- 停止当前动画
- 取消未执行动作
- 标记当前动作状态为 `cancelled`

### 12.2 失败

如果真实执行失败，例如：

- 找不到窗口
- 点击无反应
- 控件不可交互

则：

- 停止轨迹
- 标记 `failed`
- 记录失败原因
- 允许重新规划或重试

### 12.3 重试

重试前不要盲目重复，应该先：

- 重新获取坐标
- 重新验证目标窗口
- 重新生成路径

---

## 13. 延迟设计建议

为了做出“AI 正在认真操作”的感觉，延迟不能太低也不能太高。

### 默认建议

- **点击前移动**：200~400ms
- **长距离移动**：400~900ms
- **拖拽**：700~1500ms
- **动作间观察停顿**：150~300ms
- **点击波纹停留**：100~200ms

### 体验判断

- 太快：像瞬移
- 太慢：像卡顿
- 上述区间通常比较平衡

---

## 14. 可靠性分析

### 可靠性结论

**高**。

因为真实执行仍然由系统输入完成，虚拟鼠标只是视觉层，不会削弱基础可靠性。

### 可靠的原因

- Windows 桌面输入兼容性最好
- overlay 不影响真实操作
- 验证层可检测失败并回退

### 风险点

1. DPI 缩放和多显示器坐标映射
2. 焦点切换导致的坐标偏差
3. 某些窗口拦截输入
4. UAC / 管理员窗口
5. 全屏游戏 / 特殊渲染程序

这些是桌面自动化天然问题，不是虚拟鼠标独有问题。

## 15. 具体代码改造清单

### 15.1 在执行链路前增加动作调度层

当前 `Executor` 里的 `move`、`click`、`drag`、`focus_window`、`switch_window`、`shortcut` 等方法，大多是直接调用 backend 进行真实输入。要实现可见的 AI 虚拟鼠标，需要把“动作决策”和“真实执行”之间插入一个调度层。

#### 需要修改的点
- 新增 `MotionScheduler`
- 新增 `VirtualCursorState`
- 新增路径生成模块 `path.py`
- 新增 overlay 渲染模块 `overlay/renderer.py`
- 修改 `Executor` 的输入动作入口，让它先交给调度器

#### 目标
- 所有鼠标相关动作都先经过调度器
- 调度器决定动画节奏、路径和执行时机
- 真实输入仍由 backend 完成

---

### 15.2 重构 `Executor` 中的鼠标相关方法

#### 当前问题
`Executor` 中的动作是“直接调用 backend”，没有可视化前置，也没有统一的状态机。

#### 建议改造
- `move()`：改为触发虚拟鼠标移动流程
- `click()`：改为“先移动到目标点，再真实点击”
- `drag()`：改为“先移动到起点，再按下拖拽，再松开”
- `focus_window()` / `switch_window()`：增加窗口切换可视化过渡
- `shortcut()` / `type_text()`：如涉及聚焦，也要先让虚拟鼠标表达意图

#### 目标
- 保留原有 backend 输入能力
- 增加统一调度入口
- 所有动作有一致的状态和时间线

---

### 15.3 增加动作状态机

#### 需要新增状态
- `planned`
- `animating`
- `executing`
- `verifying`
- `verified`
- `failed`
- `cancelled`

#### 作用
- 让每个动作都有可追踪生命周期
- 便于调试、重试、取消和日志回放

---

### 15.4 增加虚拟鼠标状态

#### 需要维护的字段

- 当前动作 ID `currentActionId`
- 当前路径点数组 `path`
- 当前速度参数 `velocity`
- 当前状态 `state`

#### 作用
- overlay 直接读取它来渲染
- scheduler 直接更新它来推进动画

---

### 15.5 增加路径生成能力

#### 建议能力
- 贝塞尔曲线
- 分段样条
- 路径采样
- 缓动曲线 `ease-in / ease-out`
- 轻微扰动以增加自然感

#### 作用
- 避免直线匀速的机械感
- 让移动更接近人手轨迹

---

### 15.6 增加 overlay 窗口

#### overlay 特性
- 透明
- 置顶

### 15.7 增加执行后验证

#### 需要验证的内容
- 点击是否命中目标控件
- 窗口是否真正切换
- 拖拽是否发生

#### 作用
- 保证虚拟鼠标只是视觉层，不影响最终正确性

### 15.8 增加取消和中断

#### 中断来源
- 用户手动操作鼠标

- 页面状态变化导致动作失效

#### 中断后动作
- 停止动画
- 取消未执行动作
- 标记状态为 `cancelled`

## 16. 第一阶段的任务卡

第一阶段目标：先把“动作编排与虚拟鼠标状态”打通，不急着做最终视觉效果。

### 任务卡 1：定义动作与状态模型

**目标**
建立统一的数据结构，供调度器、执行器、overlay 共用。

**子任务**
- 定义 `MotionAction`
- 定义 `MotionResult`
- 定义 `VirtualCursorState`
- 定义状态枚举
- 定义动作事件结构

**验收标准**
- 任意动作都可以被表示、跟踪和回放
- 状态能完整流转

---

### 任务卡 2：抽出路径生成模块

**目标**
实现点到点移动的基础路径生成。

**子任务**
- 实现贝塞尔曲线采样
- 实现短距离和长距离不同速度策略
- 实现缓动曲线
- 支持输出离散路径点

**验收标准**
- 给定起点和终点可以生成平滑路径
- 路径点数量可配置

---

### 任务卡 3：实现动作调度器骨架

**目标**
让动作经过统一调度流程。

**子任务**
- 实现 `planned -> animating -> executing -> verifying -> verified`
- 支持失败态
- 支持取消态
- 支持动作 ID 关联

**验收标准**
- 一次 click 动作可以完整经过调度器
- 日志中能看到每个状态阶段

---

### 任务卡 4：将 click/move/drag 接入调度器

**目标**
让现有鼠标动作走新链路。

**子任务**
- 修改 `Executor.click()`
- 修改 `Executor.move()`
- 修改 `Executor.drag()`
- 保留真实 backend 输入逻辑

**验收标准**
- 旧能力不丢失
- 新链路可控

---

### 任务卡 5：设计 overlay 最小可行版本

**目标**
先把可见的虚拟鼠标渲染出来。

**子任务**
- 创建透明悬浮窗口
- 绘制鼠标本体
- 绘制轨迹尾巴
- 绘制点击波纹

**验收标准**
- 用户能在桌面上看到虚拟鼠标移动
- 不干扰系统真实输入

---

### 任务卡 6：实现执行后验证

**目标**
避免“看起来点了，实际上没点中”。

**子任务**
- 点击后读取结果
- 拖拽后检查状态变化
- 窗口切换后检查激活窗口

**验收标准**
- 动作结果可判定成功或失败
- 失败可回退或重试

---

### 任务卡 7：接入中断与取消

**目标**
允许用户随时接管控制。

**子任务**
- 监听 Esc
- 监听用户手动鼠标输入
- 取消当前动作
- 清理 overlay 动画

**验收标准**
- 任务可中断
- 中断后系统状态可恢复

### 17.9 `executor.py` 的角色调整

#### 角色变化
`executor.py` 不再是“直接执行全部动作”的地方，而是变成：

- 保留真实输入能力
- 提供动作执行原语
- 被 scheduler 调用

#### 不建议做的事
- 不要把路径动画写进 executor
- 不要把 overlay 绘制写进 executor
- 不要让 executor 直接承担状态机全部职责

## 19. 第一阶段建议的实施顺序

建议按以下顺序实施：

1. 先做动作调度器和状态机

3. 再做虚拟鼠标状态
4. 再做 overlay 渲染
5. 最后做真实输入同步和体验调优

这是最稳妥、最符合现有项目基础的实现路径。

## 21. 总结

要做出接近 Codex 的桌面交互体验，关键不是单纯把鼠标画出来，而是要把整个输入过程从“瞬时执行”升级成“可观察的动作流”。

本统一方案的核心价值是：

- 保留真实输入的可靠性
- 让用户看到 AI 的意图和轨迹
- 让动作执行变得可解释、可验证、可中断
- 在现有 Windows 桌面自动化基础上具备可落地性

这是一条现实可行、可逐步演进的路线。
